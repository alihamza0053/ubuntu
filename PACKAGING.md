# Packaging ServerHub — for Git & Hosting

How to turn this project into a clean, shareable bundle you can **push to Git**
or **upload to a host**, without leaking secrets or shipping junk.

---

## 🚀 One-click install (online) — no manual steps

Once the source is in a (public) Git repo, a **fresh Ubuntu 22.04+ server** is
set up end-to-end with a single command. It downloads the code and installs
**everything** — nginx, supervisor, MySQL, PHP, Certbot, Google Chrome, Node,
the Python venv + all pip packages — and builds the frontend. Nothing is
installed by hand.

```bash
curl -fsSL https://raw.githubusercontent.com/alihamza0053/ubuntu/main/deploy/get.sh | sudo bash
```

That's it. When it finishes, create your admin user and point your domain (the
script prints the exact commands). **Re-running the same command later updates
the panel in place.**

Override the source if your repo/branch differs:
```bash
curl -fsSL https://raw.githubusercontent.com/alihamza0053/serverhub/main/deploy/get.sh \
  | sudo REPO_URL=https://github.com/alihamza0053/ubuntu.git BRANCH=main bash
```

What the one-click does, step by step:
1. installs `git` if missing,
2. clones (or refreshes) the repo into `/opt/serverhub-src`,
3. runs [`deploy/install.sh`](deploy/install.sh) — which installs all OS
   packages + Python/Node deps and builds the UI (idempotent, safe to re-run).

> **Requirements:** the repo must be **public** (so `curl`/`git clone` can read
> it without a login). For a private repo, either clone with a token first and
> run `sudo bash deploy/install.sh`, or attach the token to `REPO_URL`
> (`https://<TOKEN>@github.com/you/serverhub.git`). Never commit the token.
>
> Prefer not to pipe the internet into a root shell unseen? Download and read it
> first: `curl -fsSL .../deploy/get.sh -o get.sh; less get.sh; sudo bash get.sh`.

## 0. What gets shipped vs. ignored

`.gitignore` is set up so these are **never** committed/bundled:

| Excluded | Why |
|---|---|
| `backend/.env` | your `SECRET_KEY` + config (secret) |
| `backend/venv/`, `frontend/node_modules/` | huge, rebuilt on deploy |
| `db/`, `*.db`, `**/backups/` | the live panel database |
| `backend/static/` | generated by the frontend build |
| `dist/` | release bundles produced by `package.sh` |
| **`/code/`, `/P2P Custom.py`** | **your private automation scripts (contain credentials)** |
| `.claude/`, editor/OS files | local tooling |

> ⚠️ **Credentials:** `code/*.py` and `P2P Custom.py` have hard-coded logins.
> They're git-ignored now, but if they were **already committed** earlier, the
> secrets are still in history — see "Scrub already-committed secrets" below.

## 1. Push to Git (the source repo)

You push the **source** (with the frontend source, so it can be rebuilt) — not
a build.

```bash
cd "Ubuntu Code Panel"        # the project root

# First time: make sure no ignored secrets are tracked
git rm -r --cached --ignore-unmatch backend/.env backend/static db code "P2P Custom.py" 2>/dev/null || true

git add -A
git commit -m "ServerHub"
git branch -M main
git remote add origin git@github.com:alihamza0053/ubuntu.git   # or https URL
git push -u origin main
```

Anyone who clones it then runs `sudo bash deploy/install.sh` to deploy.

### Scrub already-committed secrets (only if needed)
If `code/` or `P2P Custom.py` (or `.env`) were committed before, removing the
file now does **not** remove it from history. To purge:

```bash
pipx install git-filter-repo      # or: pip install git-filter-repo
git filter-repo --path "P2P Custom.py" --path code --path backend/.env --invert-paths
# then rotate any passwords that were in those files, and force-push
```

## 2. Build an upload bundle (for hosting)

When you'd rather upload a folder/archive than `git clone`:

```bash
bash deploy/package.sh
```

This builds the frontend and creates:

```
dist/serverhub/                 ← clean folder (backend + built UI + frontend src + deploy + docs)
dist/serverhub-<date>.tar.gz    ← same, as one archive
```

Neither contains `.env`, venvs, `node_modules`, databases, backups, or your
private scripts.

### Deploy the bundle on a server
```bash
scp dist/serverhub-<date>.tar.gz root@SERVER_IP:/opt/
ssh root@SERVER_IP
cd /opt && tar xzf serverhub-<date>.tar.gz
mv serverhub serverhub-src        # install.sh expects /opt/serverhub-src (or run from here)
sudo bash serverhub-src/deploy/install.sh
```
(Full server setup is in `DEPLOYMENT.md`.)

## 4. Before you publish — checklist

- [ ] `git status` shows **no** `.env`, `*.db`, `code/`, `P2P Custom.py`, `node_modules/`.
- [ ] If any secret was ever committed, history scrubbed **and** passwords rotated.
- [ ] `backend/.env.example` is present (so users know what to configure); the real `.env` is not.
- [ ] Pick a license (see `LICENSE`) — MIT for open source, or remove it / add a commercial license if selling.
- [ ] `README.md` reflects your project name / repo URL.
- [ ] `bash deploy/package.sh` runs clean and the tarball deploys on a fresh VPS.
